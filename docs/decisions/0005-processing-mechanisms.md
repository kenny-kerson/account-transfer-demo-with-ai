# ADR-0005: 이체 처리 메커니즘 (Round 2)

- Status: Accepted
- Date: 2026-05-23
- Deciders: kenny.k
- Tags: behavior, processing

## Context

`/speckit-specify` Round 2에서 이체 처리 메커니즘의 핵심 결정.

## Decision

### 타행이체 외부 금융망
- **가상 외부 시스템(Mock)을 별도 서비스로 구현**
- 실제 미들웨어처럼 호출하여 타임아웃·재시도·대사조정 등 실무 이슈를 재현
- ADR-0003 (외부 위임 시스템 구성)에서 이미 명시

### 다건이체 일부 실패 처리
- **부분 성공 허용** — 성공한 건만 이체 완료, 실패 건은 별도 상태로 격리
- All-or-Nothing 아님

### 예약이체 자금 처리
- **예약 시점에 즉시 출금·예치** — 확정성 우선
- 예약일에 추첨이 아님

### 백오피스 범위
- 이체 내역 검색·조회 대시보드
- 실시간 서비스 모니터링 (메트릭/알림)
- 이체 수동 취소·조정 (상담용)
- **제외**: 이상거래/한도 조정

## Rationale

- 다건이체 All-or-Nothing은 10,000건+ 규모에서 트랜잭션 부하·롤백 비용이 비현실적 (ADR-0007 다건 규모 결정 참조)
- 예약이체 즉시 예치는 예약일 자금 부족·계좌 폐쇄 리스크를 사전 차단 (시중은행 일반 모델)
- 백오피스에서 한도 조정·이상거래를 제외한 이유: 한도는 사용자 자기관리 + 시스템 정책으로 운영 (ADR-0006), 이상거래는 v1 제외 (ADR-0008)

## Alternatives Considered

- **다건이체 All-or-Nothing** — 거부됨. 대규모 배치(10,000건+)에서 트랜잭션 비현실적
- **예약이체 예약일 출금** — 거부됨. 자금 부족·계좌 폐쇄 등 실패 리스크 ↑
- **백오피스에 한도·FDS 포함** — 거부됨. 스코프 폭증, 별도 시스템 구축 영역

## Consequences

- (+) 다건 부분성공 → BatchTransferRequest 진행률 추적·부분 재실행 패턴 시연 가치 ↑ (ADR-0009 데이터 모델)
- (+) 예약 즉시 예치 → 예치자금 식별자·해제 정책 등 자금 관리 패턴 시연
- (+) 백오피스 범위 명확 → 권한·감사 영역 한정
- (-) 부분성공의 정합성 보장(다건 안에서 각 건이 독립 트랜잭션) 설계 복잡도
