# ADR-0003: 외부 위임 시스템 구성

- Status: Accepted
- Date: 2026-05-23
- Deciders: kenny.k
- Tags: architecture, integration

## Context

이체 서비스 코어가 어디까지 책임을 가질지가 시스템 경계 설계의 핵심이다. 모놀리식으로 모두 포함하면 코어가 비대해지고, 너무 잘게 분리하면 오케스트레이션 복잡도가 폭증한다.

사용자는 "단위 시스템 분리" 패턴(한국 금융권 표준)을 선호하여 다수의 외부 시스템에 위임하는 구조를 선택했다.

## Decision

이체 서비스는 **오케스트레이션 역할**만 담당하고, 다음 6개 영역은 **외부 단위 시스템에 위임**한다:

| # | 외부 시스템 | 책임 | 우리가 만드는 것 |
|---|---|---|---|
| 1 | 인증·전자서명 시스템 | PIN 검증, 전자서명 발급/검증 | 위임 클라이언트 |
| 2 | 계정계 | 출금·입금 원장 처리, 잔액 관리 | 위임 클라이언트 |
| 3 | 수수료 시스템 | 수수료 정책 평가, 수수료 출금 | 위임 클라이언트 |
| 4 | 타행 외부 금융망 (전금망/오픈뱅킹) | 타행 전문 송수신 | **Mock을 별도 서비스로 자체 구현** + 위임 클라이언트 |
| 5 | 알림 시스템 | SMS/Push/이메일 발송 | 위임 클라이언트 |
| 6 | (향후) FDS 시스템 | 이상거래 심사 | (v1 제외) |

타행 외부 금융망만 우리가 Mock으로 별도 서비스를 구현한다 — 실제 미들웨어처럼 호출하여 타임아웃·재시도·대사조정 등 실무 이슈를 재현하기 위함이다.

## Rationale

- 시니어 실무 수준 NFR (ADR-0002)을 만족시키려면 다층 외부 호출·서킷 브레이커·보상 트랜잭션 패턴이 자연스럽게 필요해짐
- 코어 도메인을 작게 유지함으로써 *상태머신·2-Phase 멱등·다층 락·보상 트랜잭션·다건 진행률* 같은 핵심 시연 영역에 집중 가능
- 한국 금융권 표준 아키텍처와 일치하여 학습 자료로 가치 ↑

## Alternatives Considered

- **모놀리식 (코어가 원장·인증·알림 모두 포함)** — 거부됨. 시연 가치 낮고 분리 패턴 학습 불가
- **전부 외부 위임 (Mock도 만들지 않음)** — 거부됨. 타행망의 비동기/실패/대사조정 시뮬레이션 불가능
- **모든 외부 시스템에 Mock 구현** — 거부됨. 데모 범위 과다, 핵심에 집중 X

## Consequences

- (+) 코어 도메인 축소 → 시니어 패턴 시연에 집중 가능
- (+) 외부 호출 클라이언트 다수 → 서킷 브레이커·재시도·타임아웃·보상 패턴 시연 풍부
- (+) 타행 Mock 서비스가 별도라 네트워크 실패·지연 시뮬레이션 자연스러움
- (-) 다수의 위임 호출 = 분산 트랜잭션 복잡도 ↑ → ExternalCallLog·CompensationLog 등 추적 엔터티 필수 (ADR-0009 이후 데이터 모델로 표현)
