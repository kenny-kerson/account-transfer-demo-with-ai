# ADR-0004: 이체 종류·사용자·인증 (Round 1)

- Status: Accepted
- Date: 2026-05-23
- Deciders: kenny.k
- Tags: scope, users, auth

## Context

`/speckit-specify` Round 1에서 결정된 기본 범위와 사용자·인증 모델.

## Decision

### 이체 종류 v1
- **포함**: 당행이체 / 타행이체 / 예약이체 (당행·타행) / 다건이체
- **향후 확장**: QR이체, SMS이체 (v1 제외, 식별만)

### 사용자
- **개인 고객 (B2C)** — 주 사용자
- **백오피스 사용자 (기획자·상담직원)** — 모니터링·조회·수동 조정 (상세 범위는 ADR-0007)
- 법인/사업자(B2B)는 제외

### 인증
- v1: **PIN(간편) 기반**
- 향후: 생체인증 등 단계적 확대
- **인증 시스템은 별도 단위 시스템에 위임** (ADR-0003). 우리는 인증·전자서명 요청/처리의 **클라이언트** 측을 구현

## Rationale

- v1 4종(당행/타행/예약/다건)이 시니어 패턴 시연(동시성·트랜잭션·분산락·보상·배치)에 충분히 풍부한 케이스 제공
- QR·SMS는 본질적으로 다른 인터페이스(2D 바코드 디코딩, SMS 게이트웨이 연동)라 분리하는 게 깔끔
- 인증은 도메인적으로 독립된 영역이라 위임이 자연스러움

## Alternatives Considered

- **v1에 QR·SMS 포함** — 거부됨. 스코프 과다, 핵심 시연 가치 분산
- **법인 고객 포함** — 거부됨. 결재 라인·권한 위계·다중 승인 등 복잡도 폭증
- **공동인증서 수준 강력 인증 v1** — 거부됨. 데모 환경에서 과잉, PIN으로도 인증 클라이언트 패턴 시연 충분

## Consequences

- (+) v1 범위 명확, 향후 확장 경로(QR/SMS/생체) 식별됨
- (+) B2C 단순화로 권한 모델 단순 유지 + 백오피스로 운영 시연 별도 트랙 확보
- (-) 인증 위임 클라이언트의 비동기/실패 처리가 모든 이체 path에 영향 → 설계 복잡도 ↑
